AEM OVERVIEW

AEM, a Content Management System (CMS), is employed for crafting and launching commercial websites. Its framework empowers users to precisely manage content on public sites, effortlessly manipulate site content, and cache content for expedited delivery.

1] Standard AEM Configuration


The quantity of AEM instances may differ across environments, yet a complete AEM setup generally includes

  1. Author instance
  2. Publish instance
  3. Dispatcher
  4. Load Balancer.

Each component has a specific function in managing and facilitating access to user-created content. In numerous scenarios, particularly in production, multiple publish instances and dispatchers are employed to ensure redundancy, enhance performance, and accommodate heightened traffic to the site.
Setting up an AEM instance can be done in two ways:

2] AEM on Prem:

  1. Involves significant costs for licensing and infrastructure management.
  2. Requires vigilant ownership as clients manage most aspects.
  3. Upgrades is a challenging task.
  4. Code deployment can utilize Continuous Integration (CI) tools like Jenkins, Bamboo, ACM, etc.
AEM as a Cloud Service: Operates as Software as a Service (SaaS), primarily managed by Adobe. Features an architecture that auto-scales to accommodate additional instances as needed. Reduces ownership costs since server, space, and upgrades are handled by Adobe. Adobe recommends using their managed CDN, with client CDN pointing to it. Cloud Manager, a required CI tool, adheres to a standard dispatcher configuration structure.

3] AEM Modules:

AEM Sites:

Enables the authoring and management of content pages.

AEM Assets:
Facilitates the authoring and management of assets.

AEM Forms:
  • Provides the capability to create, update, publish, and manage forms at runtime.
  • Allows adding, updating, and deleting fields in AEM Forms.
AEM Headless:
  • Utilizes an architecture that separates frontend and backend applications.
  • Exposes data through the backend in predefined formats (JSON, XML, etc.) for consumption by third-party frontend applications.
  • Achieved through GraphQL queries, Content, and experience fragments
